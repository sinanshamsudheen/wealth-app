# 06 — API Design

## URL Convention

All API endpoints follow a consistent pattern:

```
/api/v1/<module-slug>/<resource>[/<id>][/<sub-resource>]
```

### Examples

```
GET    /api/v1/engage/clients              # List clients
POST   /api/v1/engage/clients              # Create client
GET    /api/v1/engage/clients/:id          # Get client by ID
PUT    /api/v1/engage/clients/:id          # Update client
DELETE /api/v1/engage/clients/:id          # Delete client
GET    /api/v1/engage/clients/:id/interactions  # List client interactions

GET    /api/v1/deals/deals                 # List deals
POST   /api/v1/deals/deals/:id/evaluations # Add evaluation to deal

GET    /api/v1/admin/users                 # List org users
POST   /api/v1/admin/roles                 # Create role

GET    /api/v1/insights/reports            # List reports
POST   /api/v1/insights/reports/:id/run    # Execute a report
```

## Authentication

### Auth Endpoints (No token required)

```
POST   /api/v1/auth/login           # Email + password → JWT + refresh token
POST   /api/v1/auth/refresh         # Refresh token → new JWT
POST   /api/v1/auth/logout          # Invalidate refresh token
POST   /api/v1/auth/forgot-password # Send password reset email
POST   /api/v1/auth/reset-password  # Reset password with token
```

### JWT Structure

```json
{
  "sub": "user-uuid",
  "tenantId": "org-uuid",
  "email": "user@example.com",
  "roles": ["advisor"],
  "permissions": ["module:engage:access", "module:engage:clients:write", ...],
  "moduleAccess": ["engage", "deals", "insights"],
  "iat": 1712700000,
  "exp": 1712703600
}
```

### Auth Headers

```
Authorization: Bearer <jwt-token>
```

All `/api/v1/*` routes (except `/auth/*`) require a valid JWT.

## Request/Response Envelope

### Success Response

```json
{
  "success": true,
  "data": { ... },
  "meta": {
    "requestId": "req_abc123"
  }
}
```

### List Response (Paginated)

```json
{
  "success": true,
  "data": [ ... ],
  "pagination": {
    "page": 1,
    "pageSize": 25,
    "totalItems": 142,
    "totalPages": 6,
    "hasNext": true,
    "hasPrev": false
  },
  "meta": {
    "requestId": "req_abc123"
  }
}
```

### Error Response

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Client email is required",
    "details": [
      { "field": "email", "message": "Required" }
    ]
  },
  "meta": {
    "requestId": "req_abc123"
  }
}
```

### Standard Error Codes

| HTTP Status | Code | Description |
|------------|------|-------------|
| 400 | `VALIDATION_ERROR` | Invalid request body/params |
| 401 | `UNAUTHORIZED` | Missing or invalid JWT |
| 403 | `FORBIDDEN` | Valid JWT but insufficient permissions |
| 404 | `NOT_FOUND` | Resource does not exist |
| 409 | `CONFLICT` | Duplicate or conflicting state |
| 422 | `UNPROCESSABLE` | Request understood but can't be processed |
| 429 | `RATE_LIMITED` | Too many requests |
| 500 | `INTERNAL_ERROR` | Server error |

## Pagination

All list endpoints support cursor-based or page-based pagination:

### Page-Based (Default)

```
GET /api/v1/engage/clients?page=2&pageSize=25
```

### Query Parameters

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `page` | integer | 1 | Page number |
| `pageSize` | integer | 25 | Items per page (max 100) |
| `sort` | string | `created_at` | Sort field |
| `order` | string | `desc` | Sort order: `asc` or `desc` |

## Filtering

List endpoints support field-based filtering:

```
GET /api/v1/engage/clients?status=active&assigned_to=user-uuid
GET /api/v1/deals/deals?stage=evaluating&deal_type=equity
GET /api/v1/tools/tasks?priority=high&status=todo
```

### Search

Full-text search via `q` parameter:

```
GET /api/v1/engage/clients?q=john+doe
```

## Real-Time APIs

### WebSocket (Socket.IO)

Used for features requiring real-time updates:

```
Connection: wss://api.invictus.ai/ws
Auth: token sent in handshake query param
```

### Channels

| Channel | Purpose | Module |
|---------|---------|--------|
| `chat:${conversationId}` | AI chat message streaming | Platform |
| `agent:${runId}` | Agent run step updates | Platform |
| `notifications:${userId}` | In-app notifications | Tools |
| `alerts:${tenantId}` | Portfolio alerts | Insights |

### Server-Sent Events (SSE)

Used for AI chat streaming responses:

```
GET /api/v1/platform/chat/stream?conversationId=xxx
Accept: text/event-stream
```

## API Versioning

- Version is in the URL path: `/api/v1/...`
- Breaking changes require a new version (`/api/v2/...`)
- Non-breaking additions (new fields, new endpoints) don't require versioning
- Deprecated endpoints return `Deprecation` header with sunset date

## API Documentation

- **OpenAPI 3.1** spec auto-generated by FastAPI from Pydantic schemas and router definitions
- **Swagger UI** served at `/docs` (built into FastAPI)
- **ReDoc** served at `/redoc` (built into FastAPI)
- Each module's `APIRouter` contributes its own tagged section to the spec
- Pydantic `model_config` with `json_schema_extra` for example values in docs

## Rate Limiting

| Tier | Rate | Scope |
|------|------|-------|
| **Default** | 100 req/min | Per user |
| **Auth endpoints** | 10 req/min | Per IP |
| **AI/Chat** | 30 req/min | Per user |
| **Report generation** | 5 req/min | Per user |
| **Bulk operations** | 10 req/min | Per user |

Rate limit headers included in every response:
```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 87
X-RateLimit-Reset: 1712700060
```
